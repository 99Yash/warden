# CONTEXT — Warden ubiquitous language

The noun dictionary for Warden. The codebase, the docs, and any planning session should reach for these terms before inventing new ones. When language drifts in a session, sharpen the entry here. When a new concept earns a name, add it here. Don't bury definitions in code comments.

This file is **not** an architecture overview (see [`CLAUDE.md`](./CLAUDE.md)), a "why" record (see [`decisions.md`](./decisions.md)), or a milestone plan (`m{N}-plan.md`, [`scaffolding-plan.md`](./scaffolding-plan.md)). It is the layer underneath all of those.

**Conventions.** `[deferred]` marks terms that name concepts not yet built — useful shorthand, not yet load-bearing. `→ ADR-NNNN` points to the canonical justification. `→ M{N}-plan` points to a milestone plan where the term is operationalised. Type names and code symbols stay in `code style`; domain concepts stay in **bold**.

## 1. Pipeline + verbs

**`warden check`** — Fast, deterministic-only review (TSC + ESLint + npm audit + OSV verification). No LLM call. Pre-commit / CI gating. → ADR-0011.

**`warden review`** — Full pipeline: deterministic checks + context selection + LLM formatter that triages findings, writes citations, and orders **comments** by **priority order**. → ADR-0011, ADR-0012.

**`warden init`** — Builds (or refreshes) the embedding-backed context index. Three sequential **phases**: walk → chunk → embed. Idempotent re-runs hit the cache. Flags: `--rebuild`, `--dry-run`, `--max-cost`. Post-M16, `runInit()` is a thin wrapper over **`reconcileFiles`** (full-walk scope, unbounded budget); the same primitive powers `warden review`'s **incremental refresh** via the M16 alignment per ADR-0032. → ADR-0019, ADR-0032, m6-plan.md.

**`warden setup`** — `[M17]` User-level setup and readiness checks. Creates `~/.config/warden/config.jsonc` (0644) and `~/.config/warden/env` (0600) when missing, reports provider readiness for `check` / `review` / `init`, and never prompts for or prints secret values. `warden setup --check` is read-only diagnostics. `warden setup project` additionally creates the repo-level `warden.jsonc` override. Distinct from **`warden init`**, which stays repo-level and index-focused. → ADR-0034, m17-plan.md.

**`warden patrol`** — `[deferred]` Reserved verb for watch-mode (IDE-style background scanning). → ADR-0011.

**`warden security`** — `[deferred, M18]` On-demand deep SAST verb per the security-depth-tiers memory. Invokes the dedicated **security harness** with Opus 4.7 apex boss + per-file Sonnet investigators + Haiku classifiers; six-phase pipeline (Det Priors → Triage Gate → Boss Plan → Worker Fan-Out → Boss Synth → Citation Verify); substring-verifier replaces LLM-judges-LLM revalidation per ADR-0015. ADR-0029 picked **both shapes with different semantics** rather than one or the other: `warden security` is the focused verb (runs only the M18 harness; security-only output), while `warden review --deep` is a flag on `warden review` that runs the M14 review harness _plus_ the M18 security harness and unions their outputs. Originally scheduled as M14; ADR-0030 redirected M14 to the default-review harness, ADR-0031 redirected M15 to review-eval, and ADR-0032 redirected M16 to init/review alignment. → ADR-0028 §10, ADR-0029, ADR-0030, ADR-0032, m18-plan.md, memory: `project_warden_security_depth_tiers.md`.

**`warden index export` / `warden index import`** — `[deferred]` CLI verbs for bulk index migration (laptop switching, CI cache, hosted-mode handoff). Storage interfaces ship in M6; verbs ship when a concrete consumer materialises. → ADR-0016.

**review pipeline** — The phase order inside `warden review`. Three phases since the M14 close-out (2026-05-16, ADR-0030): **Phase 1 — Det Priors** (`runDetPriors()` runs the 9 deterministic runners — TSC, user-config ESLint, ESLint security, jscpd, vuln/OSV, scalability detector, consistency detector, deadcode detector, leverage detector — plus the M5/M6 context selector in parallel via `Promise.all`; reused by `warden check` directly). **Phase 2 — boss loop** (Opus 4.6 `streamText` tool-use loop with the **`dispatch_worker`** tool, capped at `stopWhen: stepCountIs(WARDEN_REVIEW_BOSS_ROUNDS ?? 5)`; dispatches the 6 review-concern workers — correctness, scalability, consistency, security, committability, leverage — per-file). **Phase 3 — Citation Verify** (the existing M10/M11 substring-verifier post-pass, moved INTO the harness as a proper phase during close-out). The M8 orchestration spine (dispatch + scratchpad + synthesizer) retired for review-mode in the same commit; `Runner` contract survives for future Phase 1 det-prior additions. → ADR-0008, ADR-0023, ADR-0030, m14-plan.md.

**phase** — A named segment of `warden init` (walk / chunk / embed) or of `warden review` (the runners + formatter steps). Each phase emits progress output and, where relevant, partial snapshots for crash recovery.

## 2. Findings, comments, citations

**finding** — A tool-verified defect from a deterministic runner (TSC, ESLint, npm audit + OSV, jscpd, M7 detectors). Findings are facts: every finding carries a source. Maps to a **comment** of `kind: "assertion"`. → ADR-0008.

**question** — An open-ended clarification or suggestion emitted by the LLM (or an LLM **sub-agent**) when intent is ambiguous. Maps to a **comment** of `kind: "question"`. May carry citations; citations are checked by the **substring-verifier** post-pass. → ADR-0021.

**`Comment`** — The unit of review output. Carries `id`, `file`, `lineStart` / `lineEnd`, `tier`, `category`, `kind`, `claim` / `body`, `explanation`, `suggestedAction`, `sources[]`, `confidence`. → ADR-0010.

**`CommentSet`** — Wrapper returned by `review()`: `comments[]` + metadata (`durationMs`, `degradedWorkers[]`). The schema is the API every future bot wrapper consumes. → ADR-0010, ADR-0013.

**`ReviewInput`** — Sole input shape for `review({ diff, repoRoot, config })`. Anchors the **I/O-pure core** invariant.

**tier** — Severity classification on a comment: **Tier 1** must-fix (production risk, vulnerability, data corruption — always post); **Tier 2** should-fix (real bug, in-scope); **Tier 3** informational (correct but low-impact or larger refactor — post only on request).

**category** — What kind of concern a comment represents. Schema-backed categories today include `correctness`, `clarity`, `style`, `dedup`, `tests`, `security`, `vulnerability`, `contract`, `scalability`, `consistency`, `deadcode`, `committability`, and `leverage`; not every slot has a producer. M12 added the `leverage` producers (this hand-rolled code duplicates a library or stdlib primitive — ADR-0027). M13 added the first `security` producers — the **ESLint security detector** half and the **security sub-agent** half (ADR-0028). Categories drive prompt shape, worker routing, and the feedback signal used for category promotion. → ADR-0012, ADR-0020, ADR-0027, ADR-0028.

**priority order** — The reading order enforced on the output, orthogonal to tier: **correctness → security → vulnerability → contract → scalability → consistency → deadcode → committability → clarity → style → leverage → dedup → tests**. `leverage` sits before `dedup` because a library/stdlib swap can dissolve a downstream dedup finding entirely. Lower-priority findings are suppressed when a higher-priority finding would change the answer. → ADR-0012, ADR-0020, ADR-0027.

**citation discipline** — The load-bearing thesis: the LLM cannot author findings without a tool source; it can only triage and format what runners produced. M7 extends the discipline to **questions** — every quoted snippet must be mechanically verifiable. The **substring-verifier** is what enforces it. → ADR-0008, ADR-0021.

**`sources[]`** — Structured citation array on every comment. Source `type` is one of `cve` (NVD/OSV), `advisory` (GitHub Advisory), `changelog`, `documentation`, `web`, `tool` (TSC / ESLint output), `repo_convention`, `api_def` (type definition from a `node_modules/<pkg>/*.d.ts` lookup, M11+). Each source carries `id` / `url` / `title` and a `retrievedAt` ISO timestamp. → ADR-0008, ADR-0026.

**`api_def` source** — `[M11+]` Source variant carrying a type definition citation from `lookupTypeDef`. Re-uses `SourceSchema`'s M10 triple: `path` = `dts_file` (repoRoot-relative path under `node_modules/`), `line` = `line_start` of the signature, `snippet` = the signature string (single-line whitespace-normalized so the verifier can match it against a concatenated `.d.ts` window). `id` carries `${package}@${version}#${symbol}` where `package` is the **literal import path** including any subpath (e.g., `drizzle-orm/sqlite-core@0.30.0#sqliteTable`); `title` carries `${kind} ${symbol}`. The LLM does not assemble these fields — `lookupTypeDef` returns a pre-shaped `suggestedSource` object the LLM copies verbatim, eliminating partial-triple parse failures. Verified by the **API claim verifier** post-pass — concat-then-substring-match against the cited `.d.ts` window, dispatched on `type`. → ADR-0026.

**evidence** — The concrete file path + single-line range (and optional snippet) that grounds a claim in the code. Surfaced on `ContextCandidate.reasons[]` (M5) and on M7 detector findings; consumed by the substring-verifier.

**clarification question** — A specific shape of comment Warden's LLM is encouraged to emit instead of asserting when domain intent is unclear. Preserves citation discipline by refusing to invent claims. → memory: `project_warden_clarification_questions.md`.

**`degradedWorkers`** — Metadata array on `CommentSet` reporting partial failures or quality signals. M7 refactor: from `string[]` to `DegradedEntry[]` with `{ kind: "actionable" | "warning" | "info", topic, message }`. Loud only when the user can act on it (e.g., "no embeddings — run `warden init`"); silent when everything is fresh.

**confidence** — Numeric 0.0–1.0 on every comment. The LLM can preserve or lower confidence, never raise it. Current hard rules use confidence as the tertiary sort key inside priority/tier buckets; a per-category confidence gate is a deferred tuning surface, not an implemented module.

**volume cap** — Maximum comments per PR (default 5, configurable). When more findings exist, prioritise by category × confidence. Excess findings are discarded, not collapsed into "show more" — prevents nag accumulation.

## 3. Models + AI layer

**boss model** — Opus-class (`claude-opus-4-6`) via `getBossModel()` in `@warden/ai` since M14 (was Sonnet-class M1–M13). The single planning brain across the M14 **boss loop**'s 5-round `dispatch_worker` tool-use loop. Distinct from the **apex model** (M18+), which sits one tier above and is reserved for the opt-in deep-security tier. M15's calibration milestone (ADR-0031) **did not bump this seat to Opus 4.7** — Config D was never built; **PD-multi** (an unplanned follow-up on Opus 4.6) cleared every evaluable gate and shipped as the new harness default. The 4.7 escape valve remains documented in ADR-0031 §2 as a future option if dogfood evidence forces it. → ADR-0006, ADR-0030, ADR-0031.

**apex model** — `[deferred, M18]` Opus-class (`claude-opus-4-7`) accessed via `getApexModel()` in `@warden/ai`. The boss seat in the dedicated **security harness** — used at exactly two call sites: Phase 2 (Boss Plan) and Phase 4 (Boss Synth) inside `warden security` / `warden review --deep` invocations. Existing `getBossModel()` stays Opus 4.6; the apex tier never inflates default `warden review` cost. Cross-provider fallback (`getApexFallbackModel()` → `gemini-2.5-pro`) is allowed only because both apex call sites are tool-less; any Gemini structured-output branch must use `transformSchemaForGemini()`. Reserved term: do not call M18's Sonnet investigators "apex workers" — workers are still the worker tier; apex names only the boss seat in the deep harness. → ADR-0029, ADR-0017, m18-plan.md.

**worker strong** / **worker sonnet** — Sonnet-class (`claude-sonnet-4-6`) specialist via `getWorkerStrongModel()`. M14 ships the worker tier proper: the **correctness**, **scalability**, **consistency**, and **security** workers under the review harness each run as `streamText` tool-use loops with `lookupTypeDef` + `readFile` + `grepRepo` tool access (`stopWhen: stepCountIs(8)`), dispatched per-file by the boss loop. M18+ adds the **security investigator worker** under the deep-security harness on the same tier. The M4-era formatter retry path retired with the M8 spine during the M14 close-out; live consumers are now the M14 review-harness workers + the M11 cascade-engaged lookups in the boss-loop's own retry shim. → ADR-0006, ADR-0030, m14-plan.md.

**worker cheap** / **worker haiku** — Haiku-class (`claude-haiku-4-5`) via `getWorkerCheapModel()` for contract, best-practices, the M7 **committability sub-agent**, the M12 **leverage sub-agent**, the M13 **security triage sub-agent**, and the M14 **committability** + **leverage** review-harness workers (the cheap-tier half of the six concerns per ADR-0030 Q6). Cheap, sufficient for pattern-matching, triage, and classification tasks. M18+ adds the **security classifier worker** for cheap sub-tasks the deep-security boss spawns ad-hoc. → ADR-0006, m7-plan.md, m13-plan.md, m14-plan.md, m18-plan.md.

**model tier** — The strong/cheap mapping per provider. Anthropic: sonnet-4 / haiku-4. Future BYO providers will mirror the tier shape (OpenAI: gpt-4.1 / mini; Google: gemini-2.5-pro / flash).

**LLM formatter** — The single Sonnet call that takes diff + tool findings + verified CVEs + retrieved context and returns a `CommentSet`. Lives in `packages/core/src/llm/`. Prompts are externalised to `prompts/{system,user-template}.md`. → ADR-0015.

**fallback chain** — Anthropic primary → 1× retry with backoff → Google Gemini secondary. Both auth and transient failures cascade. Hard-fail if both exhaust. The M4-era `@warden/core/src/llm/cascade.ts` retired with the M8 spine during the M14 close-out; the cascade now lives inline in `packages/core/src/review-harness/boss-loop.ts`. Future bot wrappers (`apps/github-bot/`) and the M18 security harness will reimplement the same retry shape rather than depend on a shared module — by design per ADR-0030. The Gemini call site wraps requests through **`transformSchemaForGemini`** per ADR-0031 to avoid the latent numeric-enum 400-noise surface. **Tools + structured-output exception (ADR-0017 2026-05-17 amendment):** call sites that combine `tools[]` with `Output.object({...})` — the M14 boss loop and every M14 worker — do NOT register a Gemini fallback because Gemini's structured-output API rejects `tools[] + responseMimeType: 'application/json'` with a 400. For those call sites the cascade is **Anthropic primary + 1× retry, hard-fail otherwise**. Tool-less structured-output call sites (including M18 Boss Plan, Boss Synth, and no-tool classifier workers) may retain Gemini fallback with the schema adapter. **Concurrency cap at the dispatch boundary** (ADR-0033) protects the no-fallback world from quota saturation; defers (b) tool-stripped Gemini retry and (c) Gemini call restructure to future ADRs gated on `apps/github-bot/` telemetry. → ADR-0017, ADR-0030, ADR-0031, ADR-0033, m18-plan.md.

**concurrency cap** — `[fix/worker-concurrency, post-M16]` Per-tier dispatch semaphore pair (`{ strong, cheap }`) constructed once per `runReview()` invocation and acquired at `makeDispatchWorkerTool()`'s `runOneDispatch` — the single dispatch boundary both Round 0's `Promise.all` and the boss's in-loop `dispatch_worker` calls flow through. Defaults: `WARDEN_WORKER_CONCURRENCY_STRONG = 4`, `WARDEN_WORKER_CONCURRENCY_CHEAP = 8`. Provider-neutral naming so M18 + BYOLLM inherit the shape without leaking Anthropic tier slugs into the env table. Each dispatch books exactly one slot from one tier; strong-tier saturation does not starve cheap-tier workers. Tier is selected from the _resolved_ worker tier (post-`routeWorker()`), so boss-side overrides via `dispatch_worker.tier?` flow through transparently. Surface is a single aggregated `degradedWorkers` info entry per review with topic `worker-concurrency`, emitted only when `totalQueued > 0` (silent on the happy path); message shape: `"concurrency cap engaged: 12/30 dispatches queued (max wait 14.2s, total queued time 31.8s)"`. Protects the **fallback chain**'s no-Gemini-fallback exception (tools + structured-output) from getting tested under quota saturation. Lives at `packages/core/src/orchestration/semaphore.ts`; the aggregate is folded into `ReviewScratchpad` via `recordConcurrencyMetric()` / `concurrencyAggregate()`. → ADR-0033, ADR-0017.

**`transformSchemaForGemini`** — `[M15]` Provider-specific Zod-schema adapter utility at `packages/ai/src/schema-adapters/gemini.ts` per ADR-0031. Walks a Zod schema and converts numeric-literal-union entries (e.g., `z.union([z.literal(1), z.literal(2), z.literal(3)])`) to string-literal-union for the Gemini structured-output request; returns a `{ requestSchema, responseTransform }` pair so the caller can post-process the parsed Gemini response back to numbers. Fixes the latent `TierEnum` stderr-noise surface observed during M14 close-out (Gemini's structured-output API rejects numeric enums; the cascade was emitting 400s to stderr). Single-file scope; no `@warden/core` schema changes. Generic provider-adapter framework deferred to BYOLLM per ADR-0006. → ADR-0031, ADR-0017.

**extended thinking** — Anthropic's multi-step reasoning mode. Used by historical M4-era plans; no live env knob in the M14 review harness. Reserved for deep-reasoning workers later if dogfood evidence earns it.

**sub-agent** — A runner whose work is a scoped LLM call (see §5 for the canonical definition). M7 ships the **committability sub-agent** (cheap-tier), which reviews added/modified files for committability anti-patterns and emits questions subject to substring-verification.

**boss/worker orchestration** — `[spine shipped in M8 (ADR-0023) and retired for review-mode in M14 (ADR-0030); worker tier shipped in M14 for the review pipeline (ADR-0030); deep-security worker tier deferred to M18 (ADR-0029/ADR-0032); other-concern workers still deferred]` The vision-tier pipeline: deterministic execution → parallel specialist workers → boss synthesis → grading. v0 collapsed this into a single formatter call. Reintroduced incrementally: sub-agent in M7; **orchestration spine** (dispatch + scratchpad + synthesizer routing existing runners) shipped in M8 per ADR-0023 and retired during the M14 close-out for review-mode; the **first vision-tier worker tier proper** shipped in M14 inside the dedicated **review harness** — 6 worker concerns (correctness, scalability, consistency, security, committability, leverage) dispatched per-file by an Opus 4.6 **boss loop** via the `dispatch_worker` tool, each worker a `streamText` tool-use loop with `lookupTypeDef` + `readFile` + `grepRepo` tool access. M15 (ADR-0031) calibrates that boss loop via **programmatic dispatch** — Round 0 fan-out is harness-controlled rather than boss-decided. M18 will add the deep-security worker tier (per-file Sonnet investigators + Haiku classifiers under an Opus 4.7 apex boss in the dedicated security harness). Other-concern workers (adversarial critic, self-aware invariant checker, free-form prose consistency) still deferred to M19+. The word **worker** is reserved for _the worker tier_ — a specialist LLM in a multi-call pipeline; M14 ships the canonical example. The M7/M12/M13 cheap-tier LLM runners (**committability sub-agent**, **leverage sub-agent**, **security triage sub-agent**) retired during the M14 close-out; their prompts seeded the M14 worker prompts per ADR-0030 Q4. → vision.md §3, ADR-0023, ADR-0029, ADR-0030, ADR-0031, ADR-0032, ADR-0008.

**self-aware boss model** — `[deferred, M6+]` Direction where the boss model can introspect its own code/limits at runtime. The introspection surface is treated as a hardened jailbreak boundary. → memory: `project_warden_self_aware_boss.md`.

**`lookupTypeDef`** — `[M11+]` AI SDK tool descriptor. Live consumers (since the M14 close-out): all 6 **worker concerns** in the review harness — the M11 formatter and the M12/M13 sub-agents that originally consumed it retired with the M8 spine and now route through the M14 workers. Input `{ package, symbol }` where `package` is the **literal import path** (`drizzle-orm`, `drizzle-orm/sqlite-core`, `@radix-ui/react-dialog`); the resolver splits `(packageName, subpath)` internally and resolves the `.d.ts` via `exports['./<subpath>']` → `typesVersions` → direct fallback → `@types/*` fallback. Output `LookupTypeDefResult` (discriminated union on `found: boolean`; `found: true` carries a pre-shaped `suggestedSource` the LLM copies verbatim into `Comment.sources[]`; `found: false` carries `reason: "package_not_installed" | "no_types" | "symbol_not_found" | "lookup_error"`). Lives at `packages/core/src/llm/tools/lookup-type-def.ts`; resolver at `packages/core/src/api/lookup-type-def.ts`. Cap: 8 calls **per worker invocation** (`stopWhen: stepCountIs(8)`); each worker gets its own 8-call budget — not centralised. `LookupTypeDefOptions.packageSearchRoots` (M12) probes additional roots (touched workspace package directories) before falling back to `repoRoot`. Triggered by per-worker prompt instructions in `review-harness/prompts/workers/*-system.md`. → ADR-0026, ADR-0027, ADR-0030.

**review harness** — Dedicated orchestration module for `warden review` at `packages/core/src/review-harness/`. Replaced the M8 spine for review-mode in the M14 close-out (2026-05-16). Runs a 3-phase pipeline — **Det Priors → boss loop → Citation Verify** — with its own `ReviewScratchpad`, an Opus 4.6 boss model dispatching per-`(file, concern)` workers via the `dispatch_worker` tool, and the existing M10/M11 substring-verifier post-pass (Phase 3, invoked end-to-end inside the harness as of close-out). Layout: `harness.ts` (entry — `runReviewHarness()` + per-tier token aggregation + cost compute), `scratchpad.ts` (`ReviewScratchpad`), `det-priors.ts` (Phase 1 — reused by `warden check`), `boss-loop.ts` (Phase 2), `workers/{correctness,scalability,consistency,security,committability,leverage}.ts` + `workers/dispatch.ts` (worker-routing fn), `tools/{dispatch-worker,read-file,grep-repo}.ts`, `prompts/boss-system.md` + `prompts/workers/*-system.md` (6 worker prompts; 3 fresh + 3 seeded from M13/M7/M12 sub-agent prompts per ADR-0030 Q4). → ADR-0030, m14-plan.md.

**boss loop** — Phase 2 of the review harness — a `streamText` tool-use loop in `packages/core/src/review-harness/boss-loop.ts` running `getBossModel()` (Opus 4.6) with the **`dispatch_worker`** tool and `stopWhen: stepCountIs(env.WARDEN_REVIEW_BOSS_ROUNDS ?? 5)`. Each round is labeled `plan` / `adjudicate` / `synth` for render UX. The boss decides per round whether to dispatch more workers (round 2–4 `adjudicate`), then emits the final review by calling the terminal `submit_review` tool with `{ comments: Comment[] }`; the harness unwraps that SDK-validated tool input into `Comment[]`. The dynamic loop subsumes the static Plan / Worker / Synth phases the M18 deep-security harness still uses into one rolling adjudication — chosen over static dispatch per ADR-0030 Q5. M15 (ADR-0031) preserves the dynamic loop's Round 1+ structure but seeds it with a deterministic Round 0 fan-out — see **programmatic dispatch**. Lane discipline drops findings whose `path` is outside the requested `files` at the dispatch boundary. The boss-loop reimplements the Anthropic → Google cascade inline (the M4-era `cascade.ts` retired with the M8 spine). Round 0's `Promise.all` fan-out and Round 1+ in-loop tool dispatches both throttle through the per-tier **concurrency cap** at `runOneDispatch` per ADR-0033. → ADR-0030, ADR-0033, m14-plan.md.

**`dispatch_worker`** — Boss-side AI SDK tool descriptor at `packages/core/src/review-harness/tools/dispatch-worker.ts`. Input zod schema: `{ files: string[], concern: 'correctness' | 'scalability' | 'consistency' | 'security' | 'committability' | 'leverage', tier?: 'sonnet' | 'haiku', focus?: string, phase: 'plan' | 'adjudicate' | 'synth' }`. Output: `{ findings, toolCalls, degraded }`. Routes `concern` → per-concern worker fn in `workers/dispatch.ts`; enforces `WARDEN_REVIEW_WORKER_BUDGET` total cap belt-and-suspenders even if the boss ignores the budget instruction in its prompt; emits degraded entry on budget exhaustion. Per-tier semaphore (the **concurrency cap**) is acquired inside `runOneDispatch` _before_ the route call and released in a `finally` per ADR-0033. → ADR-0030 §5, ADR-0033, m14-plan.md.

**worker concern** — One of the 6 specialist roles in the review harness — `correctness`, `scalability`, `consistency`, `security`, `committability`, `leverage`. Tier defaults per ADR-0030 Q6: first four are Sonnet via `getWorkerStrongModel()` (deep reasoning needed); last two are Haiku via `getWorkerCheapModel()` (pattern-matching sufficient). Boss can override per subtask via the `dispatch_worker.tier?` field; the actual tier is captured at dispatch time on `WorkerOutput.tier` so the harness can bucket per-tier cost honestly. Each worker is a `streamText` tool-use loop with `lookupTypeDef` + **`readFile`** + **`grepRepo`** tool access, `stopWhen: stepCountIs(8)`. Each has its own system prompt in `review-harness/prompts/workers/*-system.md` — three fresh (correctness, scalability, consistency) plus three seeded from the retired M7/M12/M13 sub-agent prompts. Every concern already lives in `CategoryEnum`; no schema impact. Reserved usage: "concern" is the within-harness term ("which worker am I dispatching"); the schema-level term remains **category**. → ADR-0030 Q6, m14-plan.md.

**`readFile` tool** — Worker-side AI SDK tool at `packages/core/src/review-harness/tools/read-file.ts`. Input: `{ path: string, startLine?: number, endLine?: number }`. Path-traversal check via `resolveWithinRoot`; secret-deny list (`.env`, `*.pem`, `id_rsa`, `*.key`); gitignore-honored; hard 1000-line cap with truncation marker. Built via `makeReadFileTool({ repoRoot })`. Bounded I/O-impurity scoped to the tool implementation; surrounding harness stays I/O-pure per ADR-0013. → ADR-0030 §6, m14-plan.md.

**`grepRepo` tool** — Worker-side AI SDK tool at `packages/core/src/review-harness/tools/grep-repo.ts`. Input: `{ pattern: string, glob?: string }`. Literal-substring v0 (no regex); max 200 results; secret-deny list; gitignore-honored. Shells out to ripgrep if available, else falls back to a Node-side walker. Built via `makeGrepRepoTool({ repoRoot })`. Same I/O-impurity envelope as **`readFile` tool**. → ADR-0030 §6, m14-plan.md.

**`ReviewScratchpad`** — In-memory class at `packages/core/src/review-harness/scratchpad.ts` collecting the M14 harness's phase outputs: `{ detPriors, workerOutputs[], bossTokens, workerTokens, degraded[] }`. Different shape from the retired M8 `Scratchpad` (which held `Map<runnerName, RunnerOutput>`) because the harness's content shape — det priors + per-`(file, concern)` worker outputs + tool-call provenance — doesn't fit the generic runner shape. Methods: `recordDetPriors()`, `recordWorker()`, `recordDegraded()`, `recordBossTokens()`, `bossTokens()`, `workerOutputs()`, `flattenDegraded()`, `flattenWorkerFindings()`. WorkerOutput now carries `tier?: 'sonnet' | 'haiku'` so per-tier cost aggregation in `runReviewHarness()` is honest even when the boss promotes/demotes a worker. → ADR-0030 §4, m14-plan.md.

**det priors** — Phase 1 of the review harness — `runDetPriors({ diff, repoRoot, changed, retrievedContext })` at `packages/core/src/review-harness/det-priors.ts`. Output: `DetPriors = { findings: ToolFinding[], retrievedContext, degraded: DegradedEntry[] }`. Det-prior signal per-file is also the routing primary for M15's **programmatic dispatch** Round 0 fan-out. → ADR-0030 §3, ADR-0031, m14-plan.md.

**programmatic dispatch** — `[M15, default ON post-close-out]` Boss-loop calibration shape per ADR-0031. The harness runs a deterministic Round 0 fan-out _before_ invoking the boss's `streamText` loop: it computes substantive files via det-priors output + a `≥10 substantive lines` heuristic (excluding test / doc / `BASELINE_NOISE` paths), dispatches workers via the existing `dispatch_worker` path with `phase: 'plan'` labels, then seeds the boss's initial user message with Round 0 worker outputs under a `<round_0_outputs>` block. **Two routing modes**, both controlled by `BossLoopConfig`: (a) **plain PD** — `{ programmaticDispatch: true, roundZeroExtraConcerns: [] }` — dispatches the det-routed concern only (one worker per substantive file). Eval surfaced it as 4/6 plants vs baseline's 5/6 — the single-routed-concern restriction is too narrow. (b) **PD-multi** (the post-close-out default) — `{ programmaticDispatch: true, roundZeroExtraConcerns: ['correctness'] }` — also dispatches a correctness worker on every substantive file alongside the det-routed concern (deduped). Cleared every evaluable threshold gate at 5/6 plants + ≥1 dispatch + $0.66 per eval run. The boss's role shifts from "planner + adjudicator + synthesizer" (M14 shape) to "adjudicator + synthesizer" — Round 1+ dynamism and the existing `WARDEN_REVIEW_BOSS_ROUNDS` / `WARDEN_REVIEW_WORKER_BUDGET` env knobs are unchanged. Round 0 dispatches are capped at `WARDEN_REVIEW_WORKER_BUDGET` belt-and-suspenders. Defaults live in `BOSS_LOOP_DEFAULTS` + `applyBossLoopDefaults()` in `boss-loop.ts`; opt-out is `{ programmaticDispatch: false, roundZeroExtraConcerns: [] }`. → ADR-0031.

**examples-first prompt** — `[M15, opt-in only — strict regression in eval]` Boss prompt rewrite per ADR-0031. Replaces `packages/core/src/review-harness/prompts/boss-system.md`'s rules-based shape with 6 worked examples — each `<example>` block carries `<situation>`, `<action>` (the dispatch sequence a competent boss runs), and `<synth>` (the shape of the final Comment[]). The boss-laziness memory (`project_warden_m14_boss_laziness.md`) named "restructuring the prompt around examples rather than rules" as a candidate fix; the calibration falsified it for this fixture set. → ADR-0031.

## 4. Context selection + indexing

**`ContextSelector`** — Interface in `packages/core/src/context/`. Takes the diff + repo, returns `ContextCandidate[]`. M5 ships the **CheapSignalsSelector**; M6 adds a semantic signal on top of it.

**`ContextCandidate`** — A file proposed as relevant context for the diff, plus a discriminated `reasons[]` array. Each reason carries an `Evidence[]` of single-line ranges that justify the candidate's inclusion.

**reason** (on a candidate) — Discriminated union: `direct_importer`, `direct_import`, `same_folder`, `symbol_ref` (M5); `semantic` adds in M6. Reason kind drives weighting; evidence ranges drive what excerpt the formatter sees.

**signal** — Loose term for a single source of candidates. The four cheap signals are direct-importer, direct-import, same-folder, symbol-ref. The fifth — **semantic signal** — is M6 embedding similarity, weighted at 0.9 and intensity-scaled by cosine. → ADR-0018, ADR-0019.

**`SourceParser`** / **`TsCompilerParser`** — The DI seam for AST work in `context/parser.ts`. `TsCompilerParser` wraps the TypeScript Compiler API and is currently the only place importing `typescript`. Tree-sitter swap-ins for Python/Rust/Go drop in alongside.

**chunk** / **`ChunkRecord`** — A symbol-aware code segment extracted by the **code-chunk** library (tree-sitter under the hood). Carries `chunkHash`, `filePath`, `language`, `symbolPath` (e.g., `["ClassName", "method"]`), `startLine` / `endLine`, `content`. Chunks are the unit Warden embeds. The `filePath` field is _first-writer-wins_ annotation since M6 and **not authoritative** since M16 — use **`file_chunks`** for the canonical file→chunks mapping (ADR-0032). → m6-plan.md, ADR-0032.

**code-chunk** — The chunking library Warden adopts in M6 (`CodeChunkAdapter`). Supports TS / JS / Python / Rust / Go / Java.

**embedding** — A 1024-dim float32 vector produced by Voyage `voyage-code-3` for a chunk. Stored in the `embeddings` table keyed by `(chunkHash, lockedModelId)`.

**locked model** / **locked-model index** — The `(model_id, model_version)` pair pinned to an index in `index_meta`. Voyage SKU bumps don't auto-rebuild; the user runs `warden init --rebuild` to upgrade. Prevents silent mixing of incompatible vector spaces. → ADR-0019.

**Merkle tree** / **merkle store** — Content-addressed, append-only structure over file/dir hashes for change detection. Strictly speaking not a Merkle tree in M6, but designed to graduate into one. Lives in the `merkle` table. `diff()` returns `{ changed, added, removed }`; `removed[]` was computed-but-unread until M16 (ADR-0032) wired it into **`reconcileFiles`** so deletions are pruned authoritatively. → ADR-0016, ADR-0032.

**content-addressed** — Cache keyed by content hash (SHA-256), not by mutable IDs. Same content always maps to the same address. Applied to chunks, embeddings, file snapshots, and the `import_graph` table. → ADR-0016.

**`import_graph`** — M5 cache table mapping `(file_path, file_sha)` → JSON set of imported file paths. Immutable per row. Built from AST traversal; supports reverse-lookup queries (e.g., the deadcode detector). → ADR-0018.

**`file_state`** — M5 cache table tracking which files changed since the last run, refreshed via `git ls-files --modified --others --exclude-standard`. Drives incremental selector updates. → ADR-0018.

**`RetrievedContext`** — The structured payload handed to the formatter: changed files + selected candidates + their evidence excerpts + path-only same-folder neighbours.

**banner** / **`BannerState`** — Pre-phase line in `warden review` output communicating index health: `no-index` (chunks absent), `stale` (incremental rescan needed), `model-deprecated` (locked model EOL), `model-aged` (not the current default). M7 adds `no-embeddings` (chunks present, vectors missing). Maps into `degradedWorkers`. → ADR-0019, m7-plan.md.

**gate** — A condition that disables semantic retrieval. Open gates (no-index / stale / model-deprecated / model-aged) cause the **semantic signal** to be skipped while the cheap signals still run. → ADR-0019.

**`.warden/cache.sqlite`** — The single, gitignored, content-addressed cache file the entire system reads from and writes to. Schema lives in `@warden/db`. Safe to delete; `warden init` rebuilds.

**`type_def_cache`** — `[M11+]` SQLite table caching results of `lookupTypeDef` lookups against `node_modules/<pkg>/*.d.ts`. Compound primary key `(package, version, symbol)`; content-addressed in the sense that re-resolving the same triple in the same install state is idempotent. Positive rows carry `signature` / `kind` / `jsdoc` / `dts_file` / `line_start` / `line_end`; negative rows carry `reason`. Grows with usage, not with `node_modules/` size — <1 MB per repo typical. Cache invalidation is automatic via `(package, version)` mismatch when `npm install` bumps versions; old rows for previous versions become unreachable, no explicit prune step. → ADR-0026.

**`file_chunks`** — `[M16]` Authoritative file→chunks junction table introduced by ADR-0032. PK `(file_path, chunk_hash)`; carries `file_sha` (which version of the file produced this chunk) + `indexed_at`. Indexes on both PK columns to support the two hot queries: "which chunks belong to file X?" (`reconcileFiles()`) and "which files use this chunk?" (`semantic.ts` attribution). Replaces `chunks.file_path`'s first-writer-wins approximation as the source of truth. A chunk shared across two files produces two rows. Pre-M16 indexes are bridged via the one-shot **auto-backfill** from `chunks.file_path` (gated by `index_meta.file_chunks_backfilled_at`). → ADR-0032.

**document registry** — Conceptual name (from the production-RAG framing the user reads, _Production RAG: Document Management & Indexing Strategies_) for what `file_chunks` implements. Used in M16 docs only; not a code symbol. The blog's pattern is a `documents` table with `chunk_hashes_json`; warden's normalized-SQL shape diverges from the literal but solves the same problem. → ADR-0032.

**`reconcileFiles`** — `[M16]` Shared write-orchestration primitive at `packages/core/src/init/reconcile.ts` (ADR-0032). Input shape: `{ files: WalkedFile[], removed: string[], maxUsdBudget?, ... stores + provider + locked-model }`. Embeds outside the SQLite transaction (HTTP failure leaves the file at its pre-call state); per-file atomic DB writes; **batched orphan prune** at the call boundary (one full-table scan vs O(N) per-file). Both `runInit()` (full-walk, unbounded budget) and `det-priors.ts` (stale subset, `WARDEN_REVIEW_REFRESH_MAX_USD`-bounded) call it with different scopes. `runInit()` becomes a thin wrapper post-M16. → ADR-0032.

**incremental refresh** — `[M16]` The implicit reconciliation triggered by `warden review`'s det-priors phase whenever `merkleStore.diff()` reports `(changed ∪ added ∪ removed) > 0`. Calls `reconcileFiles()` with the stale subset within budget. No new CLI flag, no new verb — opt-out via `WARDEN_REVIEW_REFRESH_MAX_USD=0`. Deletes are unconditional + free of Voyage cost. → ADR-0032.

**refresh budget** / **`WARDEN_REVIEW_REFRESH_MAX_USD`** — `[M16]` Numeric USD cap on `reconcileFiles()` invocations from the review path. Default `0.25`; env knob in `@warden/env`. Pre-flight estimate via the existing `estimate.ts` LOC heuristic; running accountant uses actual Voyage cost (`resp.promptTokens` × `VOYAGE_MODELS` pricing). Over-budget files are skipped (subsequent files within budget still refresh); surfaced as one actionable `degradedWorkers` entry: `"context: refresh capped at $0.25 — N files skipped; run \`warden init\` for a full refresh."`Set to`0` to disable refresh entirely. → ADR-0032.

**orphan prune** — `[M16]` Reference-counted cleanup pass at the end of every `reconcileFiles()` invocation. Single SQLite transaction: `DELETE FROM chunks WHERE chunk_hash NOT IN (SELECT chunk_hash FROM file_chunks)` + `DELETE FROM embeddings WHERE chunk_hash NOT IN (SELECT chunk_hash FROM chunks)`. Cheap O(N) full-table scans at warden's scale (~50k rows = ~10ms). Batched-not-per-file: per-file orphan-prune would re-scan N times. Lives on `FileChunksStore.pruneOrphans()`. → ADR-0032.

## 5. Runners — detectors + sub-agents

**runner** — Any process under `packages/core/src/runners/` that produces findings or questions. Two flavours: **detectors** (deterministic — AST, graph, structured comparison, lockfile diff; no LLM call) and **sub-agents** (cheap-tier LLM, scoped to one category, output is citation-verified). The folder name is "runners" because every runner lives there; the precise noun for any specific one is always **detector** or **sub-agent**, never bare "runner".

**detector** — A runner that emits findings via deterministic analysis only — AST traversal (TSC, ESLint, scalability, deadcode), structured comparison (consistency), or process execution (vuln, jscpd). Detector findings carry tool-grounded `sources[]`; never LLM-quoted citations. **Use a detector when the anti-pattern set is bounded _and_ reliably structural across the ecosystems Warden runs in.** A regex match for `<<<<<<<` qualifies; a regex for `node_modules/` doesn't, because legitimate code references it.

**sub-agent** — `[deprecated, M7–M13; retired in M14 close-out, 2026-05-16]` A runner that wrapped a scoped LLM call (cheap-tier), owned one **category**, and emitted questions with citations. M7–M13 shipped three (committability, leverage-libraries, security triage) as siblings of detectors in the M8 single-formatter shape. The M14 review harness retired all three for review-mode and folded their prompts into the M14 **worker concern** entries — same category coverage, dispatched per-file by the **boss loop** rather than per-runner. The term **sub-agent** is now historical; new LLM-call runners should be called **workers** and live inside a harness. → m7-plan.md, ADR-0021, ADR-0030.

**TSC runner** — Detector wrapping `tsc --noEmit` over changed files. Maps diagnostics to `ToolFinding[]`. TS-only in v0. → ADR-0008.

**ESLint runner** — Wraps `eslint --format json`. Per-rule output mapped to `ToolFinding[]` with evidence ranges. TS / JS only. → ADR-0008, ADR-0009.

**vulnerability check** — Runs `npm audit --json`, diffs the lockfile, scans added / upgraded dependencies against OSV.dev. Advisories without an OSV record are dropped — the citation discipline path lit up in M3. M7 collapses output to a summary when the manifest wasn't touched. → ADR-0008.

**OSV verification** — Calls `POST https://api.osv.dev/v1/query` (package + version) and uses the response (affected ranges, fix versions, severity) as the canonical citation for a vulnerability comment. Universal across ecosystems. → ADR-0008.

**jscpd** — Copy-paste detection runner, scoped to `changed ∪ candidates`. Surfaces dedup findings. Loaded via `createRequire` to dodge an ESM/CJS interop bug in jscpd's `colors/safe` import. → ADR-0018.

**format-check runner** — `[deferred]` Deterministic formatter signal for `warden check` / `warden review`, if implemented. Must be read-only, diff-scoped, and mutation-guarded: prefer native check modes (`prettier --check`, `biome check`, `dprint check`); run write-only formatters only in a temp copy; snapshot the working tree before and after; fail fast if the real repo changed. Diagnostics outside touched files collapse into one `degradedWorkers` entry with topic `format-check`, never a spray of unrelated comments. Lives in **Det Priors** when built. → ADR-0037.

**ecosystem detection** — Auto-derives ecosystem and framework from marker files (`package.json`, `requirements.txt`, `go.mod`, etc.) and from dependency contents. Zero manual config. → ADR-0008.

**M7 detectors** — Three new detectors land in M7: **scalability detector** (load-then-narrow queries, sequential awaits), **deadcode detector** (unused optional params, dead branches via reverse import-graph), **consistency detector** (env-var / CLI-command / file-path claims in docs that diverge from code). → m7-plan.md.

**substring-verifier** — M7 post-pass that confirms LLM-quoted snippets actually appear in their cited files (after whitespace normalisation). Drops citations that don't match; if all citations on a comment drop, drops the comment. Logs an info-level entry on drops. → ADR-0021, m7-plan.md.

**API claim verifier** — `[M11+]` Post-pass extension of the **substring-verifier**, dispatched on `source.type === "api_def"`. Reads the cited `.d.ts` file across `line ± API_DEF_DRIFT` (30 lines either side — wider than M10's 5 because `.d.ts` signatures routinely span 10+ lines), concatenates the window, normalizes whitespace once, and substring-matches the (already single-line normalized) `signature`. The wider drift + concat-then-match is the key difference from M10's per-line algorithm — M10 stays per-line for its single-line snippet consumer; `api_def` widens for multi-line signatures. Failed matches drop the source; if all sources drop, the Comment drops. Same drop semantics + `degradedWorkers` surfacing as M10. The verifier walk dispatches on source `type`; one new branch for `api_def`, all other source types unchanged. → ADR-0026.

**leverage detector** — `[M12]` Deterministic AST detector at `packages/core/src/runners/leverage.ts` emitting `kind: "assertion"` findings for three v0 stdlib idiom-miss patterns: `JSON.parse(JSON.stringify(x))` → `structuredClone(x)` (single-arg only; multi-arg `stringify(x, replacer)` / `stringify(x, null, 2)` is intentional projection, not a deep clone); `arr.indexOf(x) !== -1` / `!= -1` / `> -1` / `>= 0` (either operand order) → `arr.includes(x)`; `arr.filter(p).length > 0` / `>= 1` / `!== 0` and `arr.find(p) !== undefined` / `!= null` → `arr.some(p)`. Findings use `ToolFinding.source = "leverage"` and carry `type: "tool"` sources with `(path, line, snippet)` evidence built from the matched call expression (whitespace-collapsed). Diff-localness via `anyAddedInRange`. Runs in both `check` and `review`. Rides the M8 `Runner` contract. Disjoint with **scalability** (`sequential-await` / `load-then-narrow` live there) and with **jscpd** (code similarity, not pattern substitution). Pattern set expands in M13+ as dogfood evidence dictates. → ADR-0027.

**leverage sub-agent** — `[M12; retired in M14 close-out, 2026-05-16]` Superseded by the M14 **leverage worker concern** — same Haiku tier, same `lookupTypeDef` tool access, prompt seeded verbatim from the M12 sub-agent's. The retired sub-agent's library-agnostic four-canonical-examples shape + workspace-aware dependency preamble + empty-deps short-circuit all carry forward into `review-harness/prompts/workers/leverage-system.md` + `workers/dispatch.ts`'s `buildDepsContext()` cache. → ADR-0027, ADR-0030.

**ESLint security detector** — `[M13]` Second ESLint invocation at `packages/core/src/runners/eslint-security.ts` (`runEslintSecurity()`) using Warden-owned `eslint@10` plus a Warden-managed flat config that loads `eslint-plugin-security` + `eslint-plugin-no-secrets` via `createRequire` from `packages/core`'s declared dependencies. Goes through ESLint's Node API directly (`new ESLint({ overrideConfigFile: true, overrideConfig, cwd: repoRoot })`) — never touches the target repo's lint binary or config; loads `@typescript-eslint/parser` for `.ts`/`.tsx` files so TS source parses without project-aware type info. Rule IDs prefixed `security/*` or `no-secrets/*` are routed to `{ category: "security", tier: 1 }` in `to-comment.ts` unconditionally. Runs in both `check` and `review`, independent of whether the target repo has its own ESLint config. v0 rule list, trimmed to five by ADR-0046 — which retired `detect-child-process` / `detect-non-literal-fs-filename` / `no-secrets/no-secrets` in favor of the **react-doctor CLI runner**'s `command-execution` / `path-traversal` / secret-leak `scan()` families (single source per family) — is (`detect-eval-with-expression`, `detect-non-literal-regexp`, `detect-pseudoRandomBytes`, `detect-buffer-noassert`, `detect-disable-mustache-escape`); known-noisy rules (`detect-unsafe-regex`, `detect-object-injection`, `detect-possible-timing-attacks`, `detect-non-literal-require`) deliberately disabled per dogfood evidence. Off-the-shelf plugins by design — no custom `@warden/eslint-plugin-security` package (per ADR-0028 alternatives). Recorded into the M8 scratchpad as its own `eslint-security` entry alongside the user-config `eslint` entry. → ADR-0028.

**react-doctor CLI runner** — `[ADR-0046; planned — react-doctor-iteration-1-plan.md]` Detector at `runners/react-doctor.ts` that subprocesses the published `react-doctor` CLI with `--json --json-compact --no-telemetry`, parsing the CLI JSON (schema mirrored locally — upstream's `JsonReport` is unpublished) into `ToolFinding[]` with `source: "react-doctor"`. Availability is JSON-parse success, not exit code: `react-doctor` exits 1 when it finds issues, and `spawnCapture` still returns stdout for parsing. One call yields visitor rules + the 42-rule `scan()` SAST suite (`raw-sql`, `nosql`, `command-execution`, `webhook-signature`, `request-body-mass-assignment`, `path-traversal`, secret-leak) **+ the base-vs-head finding delta** (`--scope changed`). **NOT dead-code** (ADR-0046 §R1): react-doctor's whole-project checks run only at `--scope full`, mutually exclusive with the delta — warden drops react-doctor dead-code and relies on its own **deadcode detector**; the `--scope full` whole-repo pass is deferred to ADR-0038 `xhigh`. The runner writes warden's pruned changed paths to the CLI's hidden `--changed-files-from` input. **Per-verb npx (ADR-0046 §R2):** `review` → `npx --yes …` (fetch-on-demand) with `--scope changed --base <ref>` (ref threaded from `diff/source.ts` through `ReviewInput`); `check` → `npx --no-install …` (no install prompt; degrade if unavailable) with `--scope lines`. When no base resolves, `review` falls back to `--scope lines` and makes no introduced-only claim. **Routing** (`to-comment.ts` `mapSeverity` keys off react-doctor's coarse `category`): Security→`{security, tier 1}` (all Security tier-1 per ADR-0028 §7, eval-gated per family), Bugs→`{correctness, tier 2}`, Performance→`{scalability, tier 2}`, Maintainability + Accessibility→`{clarity, tier 3}` (no `CategoryEnum` ripple; clarity findings post but drive no worker dispatch). The `scan()` suite is the **single source** for its families: ADR-0046 retires the three overlapping M13 **ESLint security detector** rules — but that trim **rides the default-on flip, not Iteration-1** (ADR-0046 §R3), else the default config loses 3 security families. npx-on-demand keeps the oxc native binary out of warden's install tree (ADR-0003 untouched); capability-gated, **never hard-fails `check`** — absence emits one actionable `degradedWorkers` entry; 60s timeout. The TSC / `npm audit` `spawnCapture` precedent (`runners/_shared.ts`), not the ADR-0028 in-process `createRequire` path. Findings populate the optional `ToolFinding.evidence` triple by reading the flagged line (leverage precedent; the CLI supplies no snippets) — falling back to undefined-evidence when unreadable; `confidence` is assigned at `to-comment`. **Sourced** findings, independent of ADR-0044's reasoned lane; ships behind `WARDEN_REACT_DOCTOR` (default **off**) until `review-eval` clears (ADR-0031). → ADR-0046, ADR-0045, ADR-0028, ADR-0038.

**security triage sub-agent** — `[M13; retired in M14 close-out, 2026-05-16]` Superseded by the M14 **security worker concern** — promoted to Sonnet tier (was Haiku in M13), same DeepSec-borrowed prompt structure + 10-slug vocabulary + auth-bypass subtleties section, now in `review-harness/prompts/workers/security-system.md`. The M14 worker gains access to `readFile` + `grepRepo` (M14-new tools) on top of the M13 sub-agent's `lookupTypeDef` access. Confidence-floor handling stays at `applyHardRules()` (§7) — the M14 worker emits with the same Tier 1/2/3 mapping; the floor still drops sub-`0.8` confidence security findings unless Tier 1. → ADR-0028, ADR-0030.

**security harness** — `[deferred, M18]` The dedicated orchestration spine for `warden security` and `warden review --deep` (the deep tier from `project_warden_security_depth_tiers.md`). Lives at `packages/core/src/security/`; sibling to the (then-retired) M8 orchestration spine + the M14 **review harness**, not nested inside either. Runs a six-phase pipeline (Det Priors → Triage Gate → Boss Plan → Worker Fan-Out → Boss Synth → Citation Verify) with its own dispatch + scratchpad + synthesizer. Distinct from the M14 review harness: M14 has a dynamic boss loop over 6 review-concern workers; M18 has a static **boss plan** over per-file Sonnet investigators + Haiku classifiers under an Opus 4.7 apex boss. Originally planned as M14; ADR-0030 redirected M14 to the default-review harness, ADR-0031 redirected M15 to review-eval, and ADR-0032 redirected M16 to init/review alignment. → ADR-0028 §11, ADR-0029, ADR-0030, ADR-0032, m18-plan.md.

**triage gate** — `[deferred, M18]` Phase 1.5 of the security harness — a deterministic gate at `packages/core/src/security/triage-gate.ts` between Det Priors (Phase 1) and Boss Plan (Phase 2). Reads two signals: (a) any det prior emitted a finding/candidate (ESLint security hit, vuln/OSV match), and (b) the diff touches a path matching `SECURITY_SENSITIVE_PATTERNS` (hardcoded glob list covering `**/{auth,api,routes,middleware,crypto,db,queries,migrations}/**`, lockfiles, `*.sql`, `*.env*`). If **both** signals are negative, the gate skips Phases 2–5 entirely and emits one info `degradedWorkers` entry ("Deep security analysis skipped — diff has no security-relevant content"); if **either** is positive, the gate proceeds to Phase 2 and the boss decides whether the trigger is real noise (boss can still emit `subtasks: []`). The gate is intentionally cheap-and-narrow in v0; precision upgrades land in M19+ via the init-time security inventory. → ADR-0029 §7.

**boss plan** — `[deferred, M18]` Phase 2 of the security harness — a single tool-less `generateText` call to `getApexModel()` (Opus 4.7) wrapped in the multi-provider cascade. Inputs: diff + det priors + retrieved context (M5/M6 selector) + optional worker-budget instruction from `WARDEN_SECURITY_WORKER_BUDGET`. Output: structured `Plan` (Zod-typed) carrying `subtasks[]` (each `{ kind: 'investigate' | 'classify', worker: 'sonnet' | 'haiku', files: string[], slugs[], retrievedContext? }`), `skipped_files[]`, `rationale`. Allows boss-decided file batching (one subtask can cover multiple tightly-coupled files) and boss-decided slug subsets (worker focuses on the slugs the file is plausibly susceptible to). Distinct from the M14 review harness's **boss loop** (dynamic) — M18's plan is static and one-shot. → ADR-0029 §1, ADR-0017.

**security investigator worker** — `[deferred, M18]` Phase 3 producer — Sonnet investigator running as an AI SDK v6 `streamText` tool-use loop with `stopWhen: stepCountIs(8)`. One investigator per `Plan.subtasks[]` of kind `'investigate'`. Tools: `lookupTypeDef` (M11) + `readFile` (repo-scoped, secret-deny-listed, gitignore-honored, 1000-line cap) + `grepRepo` (literal-substring v0, max 200 results, repo-scoped) — the same tool kit M14 already ships for the review-harness workers. Output: `SecurityFinding[]` (worker-internal shape) carrying source-line + sink-line `tool` sources or `api_def` sources copied from `lookupTypeDef`. Workers parallelize via `Promise.all` gated on `WARDEN_SECURITY_WORKER_BUDGET`. Lane discipline drops findings whose `path` is outside the subtask's `files`. Provider fallback follows ADR-0017's tools exception: no Gemini fallback for investigators; Anthropic failure degrades cleanly with "Gemini fallback skipped (tools required)". → ADR-0029 §5, §8, ADR-0017.

**security classifier worker** — `[deferred, M18]` Phase 3 producer — Haiku classifier running as a single-shot `generateText` call (no tools) with structured `Output.object()` output. One classifier per `Plan.subtasks[]` of kind `'classify'`. Used by the boss synthesis to disambiguate borderline cases (e.g., "is line 42 in `src/log.ts` `secrets-exposure` or `log-injection`?"); does not directly produce Comments. Cheap-tier counterpart to the investigator. → ADR-0029 §5.

**boss synth** — `[deferred, M18]` Phase 4 of the security harness — a single tool-less `generateText` call to `getApexModel()` (Opus 4.7) with `output: Output.array(CommentSchema)`. Reads the `SecurityScratchpad`'s accumulated worker outputs + det priors + classifier results; dedupes findings by `(path, line)` overlap; assigns `kind: 'question' | 'assertion'` + `tier: 1 | 2 | 3` per finding; preserves citations verbatim. Boss has no tools, so Gemini fallback is allowed, but the Gemini branch must wrap `CommentSchema` with `transformSchemaForGemini()` because `TierEnum` is numeric. Verifier catches mis-cites post-hoc. → ADR-0029 §1, §8, ADR-0017.

**`SecurityScratchpad`** — `[deferred, M18]` In-memory class at `packages/core/src/security/scratchpad.ts` collecting the security harness's phase outputs. Holds `{ detPriors, plan, workerOutputs[], tokenUsage, costUsd, degraded[] }`. Different shape from both the M8 `Scratchpad` (generic `Map<runnerName, RunnerOutput>`) and the M14 `ReviewScratchpad` (det priors + per-`(file, concern)` worker outputs) because M18's content shape — structured Plan + per-subtask investigator/classifier outputs — fits neither. The three-way split is honest: each scratchpad earned its rent for its own pipeline content. → ADR-0029 §2.

**`securityRuns`** — `[deferred, M18]` SQLite table in `@warden/db` recording each `warden security` / `warden review --deep` invocation: `(id, timestamp, mode, modelBoss, modelWorkerStrong, modelWorkerCheap, inputTokens, outputTokens, costUsd, commentsEmitted)`. `mode ∈ {'security', 'review-deep'}`. Foundation a future M19+ ADR builds the weekly USD cap + `--max-cost <usd>` flag on; v0 records but does not enforce. → ADR-0029 §9.

**known debt** — Code accepted as-is and excluded from review noise. Suppressed via an optional `.reviewbot/overlay.yaml`. Small, rarely changes, kept narrow on purpose. → ADR-0008.

**`Runner`** — `[M8; contract survives the M14 spine retirement]` The contract every runner exposes: `{ readonly name: string; run(input: RunnerInput): Promise<RunnerOutput> }`. `RunnerInput` carries `changedPaths` + `repoRoot` + optional `retrievedContext`; `RunnerOutput` carries `name` + `findings[]` + optional `questions[]` + `degraded[]` + `durationMs` + optional `error?`. Live consumers (since the M14 close-out retired the dispatch fn + scratchpad + synthesizer): the scalability + leverage detectors implement it directly inside `runDetPriors()`'s `Promise.all` block. The contract is the right shape for future Phase 1 det-prior additions; it does NOT shape the worker tier (M14 workers don't implement `Runner` — they're built around the `dispatch_worker` invocation envelope instead). Lives in `packages/core/src/orchestration/runner.ts`. → ADR-0023, ADR-0030.

**`Scratchpad`** — `[M8]` In-memory class collecting per-runner outputs during a single `runReview()` / `runCheck()` call. Internal `Map<runnerName, RunnerOutput>`; methods `record()`, `get()`, `all()`, `flatten()` (returns `ToolFinding[]`), `flattenQuestions()`, `flattenDegraded()`. Bounded by runner count (~8 today, ≤20 even in M11+); pathological-diff memory pressure is solved upstream by M9's noise filter, not here. SQLite swap-point preserved for M11+ daemon scenarios. Lives in `packages/core/src/orchestration/scratchpad.ts`. → ADR-0023, m8-plan.md.

**dispatch** — `[M8; retired in M14 close-out, 2026-05-16]` Was the function that ran registered runners in parallel via `Promise.all` and wrote their `RunnerOutput`s to the M8 `Scratchpad`. Superseded by the M14 review harness's **boss loop** + `dispatch_worker` tool, which dispatches per-`(file, concern)` rather than per-runner. The `Runner` contract survives in `orchestration/runner.ts` for future Phase 1 det-prior additions (scalability + leverage detectors still implement it); the dispatch fn itself was deleted. → ADR-0023, ADR-0030.

**synthesizer** — `[M8; retired in M14 close-out, 2026-05-16]` Was the single Sonnet/Opus LLM call that read a `Scratchpad`, flattened it to `ToolFinding[]`, and produced a `CommentSet`. Replaced the M4 formatter call site; used the same `system.md` + `user-template.md` + Anthropic → retry → Google cascade. M14 (ADR-0030) replaced it with the dynamic **boss loop**, which subsumes synthesis into the terminal `synth`-labeled round of the `dispatch_worker` tool-use loop. The M4 formatter (`packages/core/src/llm/formatter.ts`) + its prompts (`system.md`, `user-template.md`) + cache + cascade module + LlmOutputSchema all retired alongside. → ADR-0023, ADR-0015, ADR-0017, ADR-0030.

## 6. Architecture invariants

**I/O-pure core** — `@warden/core` must not import `commander` / `picocolors` / `ora`, must not call `console.log` / `process.stdout`, must not read `process.argv`, must not assume a TTY. Input via `ReviewInput`; output via the returned `CommentSet`. This is the load-bearing property that makes future bot wrappers possible without rewriting the engine. → ADR-0013.

**package boundaries** — Strict imports between workspace packages. CLI may use commander / colors / ora; core may not. Core depends on AI and DB; AI does not depend on core; env is importable from anywhere; config ships only TS configs. → CLAUDE.md "Package boundaries" table.

**Warden config** — JSONC behavior config loaded by `@warden/env` in three layers: built-in defaults, global `~/.config/warden/config.jsonc`, then project `<repo>/warden.jsonc`. Config may name provider env vars (`apiKeyEnv`) and route Anthropic / Google / Voyage, but must not contain raw secret values. Project config is explicit (`warden setup project`) and commit-safe by default. → ADR-0034, docs/environment.md.

**provider readiness** — The command-scoped answer to "can Warden run here?" `check` is always ready because it is deterministic-only (one ADR-0046 nuance: the **react-doctor CLI runner**'s `scan()` families — including the three security families migrated off the M13 ESLint security detector — degrade gracefully when react-doctor can't be fetched offline/cache-cold, surfacing a `degradedWorkers` entry rather than blocking `check`); `review` requires the configured primary LLM key (Anthropic in M17); `init` requires the configured embedding key (Voyage in M17) unless dry-run. Google Gemini fallback is additive readiness, never a blocker. → ADR-0034, docs/environment.md.

**review intensity profile** — `[deferred]` User-facing depth/cost preset for `warden review`, planned as `medium`, `high`, and `xhigh`. A profile resolves to harness internals: boss rounds, Round 0 shape, worker budget/concurrency, semantic-refresh budget, M18 deep-security inclusion, model/reasoning policy, comment volume, and short-circuit gates. `medium` optimizes for P1/P2 coverage and cost; `high` is the balanced default; `xhigh` explicitly spends for risky refactors / legacy migrations / security-sensitive changes. Low-level env vars remain escape hatches, but the profile is the normal UX. → ADR-0038.

**one-shot CLI** — No TUI, no REPL, no interactive prompts. Input via flags / stdin, output via stdout / JSON, process exits. Enables bots, CI integration, and scripting. → ADR-0014.

**dogfooding** — Personal-first development. v0 ships for one user; the OSS-quality bar holds anyway because trustworthiness is the moat. M5 and M6 PRs are reviewed by Warden itself plus Copilot for ground-truth comparison. → ADR-0001, memory: `project_warden_review_category_gaps.md`.

**dogfood-lesson intake** — The discipline by which dogfood misses (bugs warden missed that another reviewer caught) enter warden's pipeline. Three lanes per ADR-0035: **detector-shaped** (M-plan backlog entry in `docs/dogfood-backlog.md`, eventually a deterministic det-prior runner per the ADR-0021 promotion path); **LLM-only residue** (prompt delta in the relevant `review-harness/prompts/workers/*-system.md` tagged with `<!-- dogfood: <YYYY-MM-DD> <repo>#<PR> — <summary> -->`); **repo-overlay-specific** (candidate for `.reviewbot/overlay.yaml`, not warden core). Retirement discipline: when a detector ships, its corresponding prompt delta is removed in the same change so warden does not double-pay. Audit cadence is `rg "<!-- dogfood:"` until active-delta volume earns the eval-suite + persistence layer named in ADR-0035's caveats. → ADR-0035, ADR-0020, ADR-0021, memory: `project_warden_alfred_pr14_misses.md`.

**security depth tiers** — Always-on fast pass + default review + on-demand deep mode. Critical findings short-circuit through the gate regardless of mode. → memory: `project_warden_security_depth_tiers.md`.

## 7. Quality metrics

**confidence threshold** — Per-category numeric floor for silently dropping low-confidence findings before render. Implemented in `packages/core/src/confidence.ts` as `CATEGORY_CONFIDENCE_FLOOR: Partial<Record<Category, number>>` consumed by `applyConfidenceFloor()` (called from `applyHardRules()` before the Tier-3 verbose gate and the priority sort). v0 floors: `{ security: 0.8 }`; other categories implicit 0 (no filtering). **Tier-1 findings bypass the floor unconditionally** — the critical-finding short-circuit named in `project_warden_security_depth_tiers.md`. Drops surface as one info-level `degradedWorkers` entry per non-zero drop count per category (the message quotes the _effective_ floor — env override or default — not always the static default). Override: `WARDEN_SECURITY_CONFIDENCE_FLOOR` env var; other categories add their own env vars on demand when they ship with non-zero floors. **M18 carve-out (deferred):** `applyHardRules()` will gain a `{ harness: 'm14-review' | 'm18-security' }` discriminator (ADR-0029 §10); the `m18-security` branch will **skip** the confidence-floor step entirely on the principle that the deep-security epistemic structure (Opus apex boss + Sonnet investigators with tools + substring-verifier) is materially different from M13's Haiku-tier triage and the verifier is the sole gate. The M13 floor stays in place for the M14 review path unchanged. Future-tunable per category as dogfood reveals volume-control need — style is the natural next candidate (very conservative on nits). → ADR-0028, ADR-0029, ADR-0030.

**false positive rate** — % of comments developers mark "Not Useful". Prototype expectation < 35%; mature target < 25%.

**usefulness rate** — % of comments developers mark "Useful". Prototype expectation 40–60%; mature target > 70%.

**address rate** — % of comments resolved in the final commit (post-merge re-run). Prototype 30–50%; mature > 60%.

**latency** — Time from PR creation to comments posted. v0 target < 10 min; mature target < 5 min.

**multi-criteria threshold** — `[M15]` The exit gate ADR-0031 applies to candidate boss-loop calibrations evaluated against `review-eval`. Five conjunctive criteria: (a) catches ≥2 of 3 documented M14 close-out missed issues; (b) catches ≥4 of synthetic plant fixtures; (c) emits 0 comments on both clean-diff fixtures; (d) total cost <$3 per run; (e) dispatches ≥1 worker on every substantive fixture. Median-of-3 sampling per (fixture, config) suppresses LLM variance. **M15 close-out outcome:** no config cleared all 5 gates because (a) m14-closeout failed _universally_ — workers can't read modified files in the eval temp repo, so the gate is fixture-limited rather than config-limited. **PD-multi cleared every evaluable gate** (b/c/d/e) and shipped as the new default. The (a) gate gets revisited in a follow-up that adds modified-file + multi-file diff materialization to `review-eval`. Distinct from the **confidence threshold** above (per-category drop floor inside the harness) and from **false positive rate** / **usefulness rate** / **address rate** (continuous metrics measured over many runs once a feedback-loop surface exists). Implementation: `packages/cli/scripts/eval/score.mts`. → ADR-0031.

**review-eval** — `[M15]` Lean internal calibration fixture suite for the M14 review harness per ADR-0031. Lives at `packages/cli/scripts/eval/` alongside the M-plan smoke scripts. Hybrid fixture set: ~5–7 synthetic plants (one per worker concern + 2 clean controls) under `fixtures/synthetic/` + ≥1 real warden PR with hand-graded ground truth under `fixtures/real-prs/` (M14 close-out delta pre-labeled by `project_warden_m14_boss_laziness.md`; earlier PRs hand-graded in `labels.md`). Output: per-fixture pass/fail on synthetic + scorecard rows on real PRs + aggregate JSON written to `results/<timestamp>.json` + console markdown table + the **multi-criteria threshold** verdict line. CLI surface: `pnpm eval` (run all configs against all fixtures with N=3 samples per pair) / `pnpm eval --config <name>` (single config) / `pnpm eval --compare <a> <b>` (diff table). Distinct from the deferred **state-of-the-art verification suite** (§8) — review-eval is internal-tuning, not a public-claim benchmark; when SOTA ships, it absorbs review-eval as one subset rather than the other way around. The suite imports `@warden/core` directly (not via the CLI) so future bot wrappers (ADR-0013 `apps/github-bot/`) can re-use it without restructuring. → ADR-0031.

**committability detector half** — A detector for the genuinely bounded subset of committability anti-patterns (merge-conflict markers, leftover `debugger`, `// TODO(remove before commit)`-style sentinels) — i.e., the patterns that survive the "reliable across ecosystems" test. Consciously parked: the open-tail sub-agent is the v0 implementation, and splitting was rejected in M7 because the supposedly-bounded patterns (dirname conventions, hardcoded path heuristics) turned out to be context-dependent. Revisit if a clean bounded subset surfaces.

**diff-level noise filter** — `[M9 v0]` A pre-runner stage in the diff loader that prunes the diff before any runner consumes it, defending the catastrophic case where committed `node_modules/` explodes the input for _every_ runner. Composes the **`BASELINE_NOISE`** language-agnostic floor (OS / editor junk: `.git/`, `.DS_Store`, `*.pyc`, `*.swp`, `Thumbs.db`, `.vscode/.history/`) with per-ecosystem **noise profiles**. Internal representation is a **diff tree** built from `parseUnifiedDiff()` output (`ChangedFile[]` grouped by directory, file count per node) — bounded by directory structure, not file count, so 500K-file diffs fit in a few KB. Pruned subtrees each emit a `degradedWorkers` entry of `topic: "noise-filter"`, `kind: "actionable"`, naming the path + count + ecosystem. M9 v0 ships exactly one profile (`javascript.json`) + the baseline floor; no user override surface (overlay deferred to M10's own milestone); no structural fallback heuristic (M7's directory-concentration placeholder dropped — false-positive risk without overlay escape hatch). Per-subtree ecosystem detection deferred to M11+ (depends on the multi-ecosystem detector rewrite). M7 ships a single-heuristic placeholder inside `committability.ts` (>80% directory concentration _or_ >200 added files no dominator), removed in M9. Design: ADR-0022 (direction) + ADR-0025 (M9 v0 scope); implementation: `m9-plan.md`.

**`BASELINE_NOISE`** — `[M9 v0]` Language-agnostic noise constant in `packages/core/src/diff/prune.ts`, applied unconditionally before any **noise profile**. Lists OS / editor junk that's noise regardless of ecosystem: `.git/`, `.DS_Store`, `*.pyc`, `*.swp`, `Thumbs.db`, `.vscode/.history/`. Graduated from M7's `committability.ts` (where it lived as the Tier-1 hard-skip list); now applied universally at the diff loader so every runner — not just the committability sub-agent — gets the floor. Distinct from **noise profiles** (per-ecosystem) by being language-agnostic. → ADR-0025.

**noise profile** — `[M9 v0]` Per-ecosystem JSON document at `packages/core/src/ecosystem/profiles/{ecosystem}.json` listing always-noise directories and extensions. v0 schema: `{ ecosystem, alwaysNoise: { directories, extensions } }` — no `contextDependent` bucket (no escape hatch without overlay), no `files` bucket (lockfiles are diff signal, not noise — vuln runs against `repoRoot`, not the diff), no schema versioning (YAGNI). v0 ships exactly one profile (`javascript.json`); Python / Rust / Go / Java / C# / Ruby profiles defer to M11+ alongside the multi-ecosystem detector rewrite. Distinct from **`BASELINE_NOISE`** (language-agnostic floor in `diff/prune.ts`) — profiles add ecosystem-specific entries on top of the universal floor. → ADR-0025.

**diff tree** — `[M9 v0]` Depth-limited (≤3) representation of a diff, built from `parseUnifiedDiff()` output (`ChangedFile[]`) by grouping paths by directory. Each node carries `(path, fileCount, children)`. Bounded by directory structure, not file count — a 500K-file diff still fits in a few KB. Internal to `packages/core/src/diff/`; not exposed on runner contracts (β interface per ADR-0023 §5; runners consume `path[]`). The `(addedCount, modifiedCount, deletedCount)` triplet from ADR-0022 §4 reduces to `fileCount` in v0 — no consumer reads the breakdown; reintroduce when one does. → ADR-0025.

## 9. Observability + review runs

`[M-tbd, ADR-0048]` Planned vocabulary for two distinct surfaces that share one **review run** identity. Keep them separate in language: the **review trace** is the persisted audit artifact (trust spine); the **observability surface** is the live dev tool. Current shipped code is only the `docker/langfuse/` stack and `LANGFUSE_*` env scaffolding; the table, run-id threading, OTEL export, and dropped-candidate events are not implemented yet.

**review run** — `[planned]` One invocation of the review harness, identified by a **run-id**. The unit a **review trace**, the **observability surface**, and (future) resume all key off. Will be persisted as a `reviewRuns` row in `.warden/cache.sqlite` (mirroring `securityRuns`), cloud-compatible via the ADR-0016 storage interfaces. → ADR-0048, ADR-0044 §7.

**run-id** — `[planned]` The random, durable identity of a **review run** (`createId("run")`). Doubles as the Langfuse trace-grouping key so a boss + all its workers collapse into one trace tree. Minted once in `runReviewHarness()`, threaded to workers via `dispatch_worker` and to the eval harness (which tags traces by config / fixture / sample). Distinct from **input-hash** — two genuine re-runs of the same diff get different run-ids but the same input-hash. → ADR-0048.

**input-hash** — `[planned]` The content-addressed key of a **review run**: a hash over `(diff_hash, resolved config, sorted model-set)`. The dedup / resume lookup key (does a prior run of _this exact review_ exist?). Supersedes the dead `llm_review_cache` content key (ADR-0007) as the resume substrate. The per-worker resume sub-key adds `(concern, sorted dispatched files, prompt-hash, model)`. Resume itself is designed-but-deferred. → ADR-0048 §8.

**review trace** — `[planned, ADR-0044 §7]` The trust spine: a read-model derived from `ReviewScratchpad` at the end of `runReviewHarness()`, recording the _deterministic_ pipeline facts (det-priors fired, dispatches `(file, concern, tier, toolCalls[])`, per-finding `{provenance, confidence, kind, evidenceVerified, sources[].type}`, and the deterministic transforms that changed the output). **Local, descriptive, prose-free, and never a gate** — it records _how_ a review was produced, never decides _what posts_, and excludes the boss's free-text reasoning. Will be persisted on the `reviewRuns` row; a compact summary can ride `CommentSet.metadata`. → ADR-0044 §7.

**observability surface** — `[planned, ADR-0048]` The live, dev-time debugging + cross-run-diffing tool, distinct from the **review trace**. When implemented, OTEL spans (AI-SDK `experimental_telemetry` auto-instruments every LLM + tool call; exporter isolated to `@warden/ai`) ship to a **self-hosted** Langfuse stack (`docker/langfuse/`), grouped by **run-id**. Non-authoritative — it never becomes the audit artifact and never gates output. Self-hosted only (source in prompts must not leave the box); I/O capture (incl. boss prose) defaults ON via `WARDEN_LANGFUSE_CAPTURE_IO`. The prose-free invariant of the **review trace** binds the persisted audit artifact only — it does not bind this surface (ADR-0048 §6 carve-out). No keys → total no-op once the exporter exists. → ADR-0048.

**dropped-candidate event** — `[planned, ADR-0048 §4]` An explicit trace event emitted when a finding the harness _considered_ is discarded by a deterministic transform: lane-discipline path drop, uncited-finding drop, off-hunk / non-patch-line scope drop (`scopeCommentsToDiff()`), volume-cap discard, or confidence→kind degrade. The recall signal generic tool-spans miss — a dropped candidate never becomes a tool call, so emit it explicitly. Same source (the ADR-0044 scratchpad read-model), two sinks: the persisted **review trace** and live OTEL span events on the **observability surface**. → ADR-0048 §4, ADR-0044 §7.

## Open inconsistencies

Things the docs spell more than one way. Pick one before they ossify.

- **boss model** vs **boss LLM** vs **synthesiser** — the same role; prefer **boss model** in code, **boss** in prose.
- **worker strong** vs **worker sonnet** vs **strong worker** — prefer **worker strong** to mirror **worker cheap**.
- **finding** (tool-derived) vs **comment** (output unit) — never use them interchangeably; the formatter promotes findings into comments.
- **citation** vs **source** — `sources[]` is the field; **citation** is the activity / discipline. Don't say "the citations array."
- **selector** vs **context selector** vs `ContextSelector` — prefer **selector** in prose, `ContextSelector` for the type, **context selector** only when disambiguation is needed.
- **`WorkerTier` enum (`'sonnet' | 'haiku'`)** vs **strong/cheap nomenclature** — the env vars introduced by ADR-0033 (`WARDEN_WORKER_CONCURRENCY_STRONG` / `_CHEAP`) and the getter functions (`getWorkerStrongModel()` / `getWorkerCheapModel()`) use provider-neutral `strong`/`cheap`; the `dispatch_worker` tool's `TierEnum` still uses Anthropic-specific `'sonnet' | 'haiku'`. Out of ADR-0033 scope to rename — a future schema-migration ADR may collapse the vocabulary. Pick **strong/cheap** in new prose and new env/code surfaces; tolerate the `TierEnum` slugs at the tool boundary until renamed.
